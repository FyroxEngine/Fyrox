CHAPTER 1  ·  Origin & Philosophy
────────────────────────────────────────────────────────────────────────────────

1.1  THE 65-PAGE PROBLEM

Every large creative project eventually produces a document too heavy to carry.
Characters sprawl. World rules contradict each other three arcs later. Lore written
in Act I becomes invisible by the time Act III needs it. A writer sitting down to
continue work must re-read sixty-five pages of notes before writing a single sentence.

This is the problem Quantum Quill was built to solve.

The Quantum Quill is not a writing tool. It is not a note-taking app, a wikis
system, or a lore database. It is a Narrative DAW — a Digital Audio Workstation
whose medium is story instead of sound.

A music producer does not manually track every note, every layer, every effect
across a forty-eight-track session. The DAW does it. The producer composes. The
DAW manages the state. When the producer needs the bass line from Act II, they
grab it from the arrange view and drop it in. It is always there. It is always
consistent with everything around it.

Quantum Quill applies that exact philosophy to worldbuilding and narrative. Every
character, event, faction, object, and piece of lore is a Capsule — an atomic
unit that knows what it is, where it came from, and what it connects to. Capsules
live inside Containers. Containers live inside Mythos domains. Everything lives
inside the Genesis Container — the sealed, physics-locked root of the entire world.

The 65-page problem disappears when you stop writing notes and start seeding
a world engine.

1.2  THE DAW METAPHOR — WHY IT WORKS

Digital Audio Workstations solved state management for sound. Every parameter of
every sound at every point in time is tracked, versioned, and recallable. Layers
are independent but composable. Rendering (playing the audio) is separate from
the arrangement that defines it. You can change the mix without changing the
composition. You can export to any format without touching the project.

Quantum Quill applies each of these properties to narrative:

  · Every story element is tracked, versioned (via B-DNA lineage), and recallable.
  · Narrative layers (Capsules, Containers, Mythos domains) are independent but
    composable through the Wire type system.
  · Rendering (exporting to screenplay, prose, game data, or audio script) is
    separate from the world state that defines the narrative. The world runs
    headless. Output adapters observe it.
  · You can change the visual presentation without changing the story engine.
  · You can export to any format without touching the simulation.

The DAW metaphor is not decoration. It is the architectural specification.

1.3  BIOSPARK STUDIOS — THE CREATOR ENTITY

BioSpark Studios is the creative and technical entity behind the entire Quantum
ecosystem. It operates as a full creative studio — narrative, music, software,
and simulation — with all projects living inside the Quantum architecture.

The Quantum Quill is BioSpark Studios' internal operating system for creative work.
Everything made at BioSpark Studios is a Genesis Container. Every artist, world,
album, software module, and agent is seeded into the hierarchy and governed by
the Octave Capacity Law, the Wire type system, and the B-DNA lineage registry.

The fictional universe of Xyrona Prime provides the symbolic and mythic DNA for
all visual design, faction conditioning, agent behavior, and resonance physics
across the entire stack. The mythology is not separate from the technology — it
is the naming and aesthetic layer that makes the technology feel alive.

CHAPTER 2  ·  Glossary of Core Terms
────────────────────────────────────────────────────────────────────────────────

GENESIS CONTAINER
  The sealed, physics-locked root of a BioSpark project. Every world, application,
  musical artist, or simulation lives inside exactly one Genesis Container. It
  declares the base resonance frequency, the wire types active in the world, the
  growth mode, and the seed answers that defined the world's physics. A Genesis
  Container at seal time is immutable in structure. It is the canvas with its
  dimensions locked.

MYTHOS CONTAINER
  The first level of subdivision inside a Genesis Container. Mythos Containers
  represent major domains: a story arc, a world region, an album series, a software
  department, or an agent guild. They each declare their own octave depth and growth
  mode, independent of the Genesis Container's settings.

CONTAINER
  The second subdivision level. Containers hold Capsules. In narrative terms, a
  Container might be a chapter collection, a culture, or a faction. In music, it
  is an album. In software, it is a feature set. Containers are the organizational
  workhorse of the hierarchy.

CAPSULE
  The atomic unit of everything. A Capsule is a single entity: one character, one
  event, one lore entry, one track, one component, one agent. Every Capsule carries
  a wire type (what kind of data it carries), a B-DNA lineage hash (where it came
  from), a resonance frequency (its place in the Harmonic Tensor Graph), and a
  heraldic birth position (its symbolic identity at creation). Capsules cannot be
  subdivided further.

VAULT
  The persistent storage and publication system for all completed or shared
  BioSpark artifacts. Blueprints, Modules, Collections, Tomes, and Revisions
  are the Vault's primary data model. The Vault is the delivery layer — the
  Genesis Container is the creative layer.

WIRE TYPE
  The signal type carried by a connection between nodes. There are 17 wire types
  in the current canonical system (see Chapter 7). Wire types are strictly enforced
  — a DATA wire cannot carry AUDIO data without an explicit conversion node.

B-DNA
  Biographical DNA. The lineage record attached to every Capsule and Container that
  tracks its origin, generation, parent IDs, and any covenant bindings. The B-DNA
  system answers three questions: Where did this come from? What has it evolved
  into? What rules follow it wherever it goes?

SEAL / SEALING
  The act of locking a Genesis Container's structure. After sealing, the hierarchy
  is immutable. Capsule payloads may update, but no new Containers can be added and
  no existing ones removed. Sealing is irreversible. The world's physics are frozen.

PORTAL
  The user-facing access point for interacting with the Quantum Quill. Where the
  DAW's arrange view meets the user's creative intent.

STUDIO SCRIBE
  The AI layer that translates natural language input into Capsule seeds, B-DNA
  records, and wire connections. The Scribe listens. The engine acts.

LOOM ENGINE
  The five-stage narrative composition pipeline: Warp Threads → Pattern Harnesses
  → Operation Shuttle → Reed Merger → Take-Up Roll. The Loom is the part of the
  engine that turns discrete Capsule events into flowing narrative output.

BLOOM EFFECTS
  The emergent narrative events that arise from Capsule interaction — events the
  author did not explicitly program, which the engine surfaces from the state of
  the world. Bloom is the difference between a story engine and a story database.

OCTAVE
  A power-of-2 capacity level. Octave 4 = 16 children (canonical default).
  Octave 5 = 32. Octave 3 = 8. See Chapter 13 for the full law.

RESONANCE
  Every node in the system — Genesis Container, Mythos, Container, Capsule, Module
  — declares a resonance frequency in Hz. These frequencies form the Harmonic Tensor
  Graph, which governs dynamic edge formation, gate condition evaluation, and heraldic
  position calculation. 440Hz is the Aethel-Vora handshake frequency. 432Hz governs
  the Forge/soul-weight system. 88Hz is the Sky-Surface trade (the Holy Hum).

FACTION
  One of five world-agnostic aesthetic presets (Luminarite, Venturan, Sylvanid,
  Hydralis, Syntaran) that can be applied to any BioSpark project to condition its
  visual design, emotional register, and AI generation output.

CREST
  A Level 2 heraldic symbol representing dynastic identity. Crests are assigned to
  major systems, departments, or domains. There are 11 named Crests (Core, Atlas,
  Vault, Mythos, Codex, Loom, Composer, Forge, Order, Mind, Soul) plus 5 reserved.

CHAPTER 3  ·  Design Principles
────────────────────────────────────────────────────────────────────────────────

3.1  EMERGENCE OVER PRESCRIPTION

The engine does not tell stories. It runs worlds. Stories emerge from the state of
the world as actors make decisions, relationships form under social pressure, and
resources shift between factions. The author seeds the world — the world tells
its story.

This principle has a direct technical implication: the narrative engine must run
headless. It must produce events, evolve state, and generate Bloom Effects with
no renderer attached. If you cannot run the world from the command line and see
it producing events, the engine is not designed correctly. Rendering is a
consequence of the simulation. It is never the host.

3.2  MODULARITY WITHOUT COUPLING

Every Quantum Module is a standalone library crate. It communicates with other
modules exclusively via WirePackets through the Theater compositor. A module never
imports another module's types. A module never calls another module's functions.
They are strangers to each other — neighbors who communicate only through the mail
system they share.

This is not an abstract principle. It is enforced at the Cargo dependency level.
If adding Module A as a dependency of Module B does not cause a compilation error,
the isolation is accidental. The correct state is that it IS a compilation error,
because Module A is never in Module B's dep tree.

3.3  USER AGENCY — THE CANVAS BEFORE THE PAINT

Every container declares its constraints before content enters it. Growth mode
(Fixed or Dynamic), octave depth, resonance frequency, and wire types are seed
decisions — made once, at the beginning, before any Capsules are added. This is
not a limitation. It is the canvas size. A painter chooses their canvas before
picking up a brush, not after the painting is half-finished.

Fixed containers provide creative constraints intentionally — the boundary IS the
point. Dynamic containers grow as the work demands — the shape is allowed to emerge.
Both modes are valid. The choice is the design decision.

3.4  PROVENANCE IS MANDATORY

Nothing without provenance enters a sealed Genesis Container. Every Capsule must
have a B-DNA lineage hash. Every derivative work must declare its parentIds. Every
covenant binding must specify its scope (local, inherited, or cross-context).

This is not bureaucracy. It is the answer to the 65-page problem. When you need
to know whether the lore event in Act III contradicts the physics established in
Act I, you do not re-read sixty-five pages. You follow the B-DNA chain.

3.5  LOCAL-FIRST ALWAYS

All inference, storage, and state management defaults to local hardware. Ollama is
the primary LLM provider. SQLite and ChromaDB are the default storage systems.
Firebase, Firestore, and cloud services are only considered when there is absolutely
no local alternative that meets the requirement.

This principle applies universally — to agent routing, to model selection, to vector
search, and to data persistence. The system must function without internet access.
Cloud is a feature, not a requirement.

3.6  ENGINE DEFINES CONCEPTS. WORLDS DEFINE CONSTANTS.

The narrative engine is world-agnostic at the code level. It defines resonance as a
concept, factions as a concept, and sacred frequencies as a concept. It does not
hardcode what any of these mean.

A world running on this engine declares its own constants in its SeedAnswers — what
frequency is sacred, what factions exist, what the balance law looks like. The engine
reads these values and operates on them without caring whether they came from Xyrona
Prime or a hard-sci-fi megastructure simulation.

  Engine-level (always present, always agnostic):
    · WireType enum (17 types)
    · BDna([bool; 64]) — the lineage mechanism
    · Octave Capacity Law — the capacity math
    · Four hierarchy levels
    · FactionPreset struct — the concept of a faction
    · WorldConstants struct — the concept of sacred values
    · CovenantBinding — the concept of a world law

  World-level (declared in SeedAnswers and world seed files):
    · The names and colors of factions
    · The frequencies that are sacred
    · Which balance laws are active
    · What the agent personas are called
    · What the visual divinity specs look like

Xyrona Prime is the canonical default world seed. It ships as xyrona-prime.qgcp —
a fully seeded Genesis Container with the Five Factions, the three sacred frequencies
(440/432/88 Hz), and RULE-EQUIL-02 pre-defined as a CovenantBinding. Any world that
wants a head start can load this seed. Any world that doesn't is free to ignore it
entirely without fighting any engine-level assumptions.

This means the constant declarations in Appendix E are Xyrona Prime's constants, not
the engine's. They live in the world seed file. The engine code has no `const
AETHEL_VORA_HZ`. It has `world.constants.handshake_hz`.

CHAPTER 4  ·  The Three-Layer Architecture
────────────────────────────────────────────────────────────────────────────────

The myth-os system — the Rust engine beneath the Quantum Quill — is organized
into exactly three layers. Violating the boundary between any two layers has caused
full project restarts. The rules are not flexible.

┌──────────────────────────────────────────────────────────────────────────────┐
│  LAYER 1 — NARRATIVE ENGINE                                                  │
│  crates: myth-wire, mythos, qgcp                                             │
│                                                                              │
│  Pure Rust. Zero renderer dependencies. Runs headless.                       │
│  Owns: world state, actor simulation, wire protocol,                         │
│  genesis container hierarchy, B-DNA, tick loop.                              │
│                                                                              │
│  Compile check: cargo build -p mythos                                        │
│  must succeed with ZERO renderer crates in the dep tree.                     │
│  Run: cargo tree -p mythos | grep bevy → must return nothing.                │
└──────────────────────────────────────────────────────────────────────────────┘
           │  emits WirePackets, exposes WorldState
           ▼
┌──────────────────────────────────────────────────────────────────────────────┐
│  LAYER 2 — MODULES                                                           │
│  crates: one per Quantum module (terrain, loom, chronicle…)                  │
│                                                                              │
│  Each module is a standalone library crate.                                  │
│  Depends on: myth-wire + its own domain libs only.                           │
│  Has a fn main() so it can run standalone.                                   │
│  Communicates with other modules via WirePackets ONLY.                       │
│  Never imports another module crate directly.                                │
└──────────────────────────────────────────────────────────────────────────────┘
           │  modules emit/consume WirePackets
           ▼
┌──────────────────────────────────────────────────────────────────────────────┐
│  LAYER 3 — OUTPUT ADAPTERS                                                   │
│  crate: biospark-theater (with Cargo feature gates)                          │
│  bins:  engine, instruments, theater-bin                                     │
│                                                                              │
│  Adapters read world state and module output.                                │
│  They have NO write access to the narrative engine.                          │
│  They are entirely optional — the world runs without them.                   │
│  Renderer deps live HERE only (bevy, egui, cpal).                            │
└──────────────────────────────────────────────────────────────────────────────┘

4.1  LAYER 1: NARRATIVE ENGINE

The narrative engine is the only layer that knows the world exists. It holds world
state, runs the actor simulation, enforces the Octave Capacity Law, manages B-DNA
records, and emits WirePackets describing everything that changes.

Three crates live in this layer:

  myth-wire   — The type system. WireType enum (17 variants), WirePacket,
                WirePayload, BDna (Vec<bool>, exactly 64 elements), ModuleId,
                ChannelId. Pure types. No logic. No async runtime. Deps: serde
                and bincode only. No other deps are permitted, ever.

  mythos      — The simulation core. World state types, actor simulation logic
                (instinct, memory, social drive, dialogue), Genesis Container
                hierarchy, Octave Capacity Law enforcement, and the tick loop.
                Deps: myth-wire, serde. No renderer. No async runtime. No
                module-specific logic (that lives in Layer 2).

  qgcp        — The file format layer. Defines .qgenesis and .qgcp serialization.
                Deps: myth-wire, serde. This crate is how worlds are saved,
                loaded, packaged, and distributed.

4.2  LAYER 2: MODULES

There are 16 Quantum Modules, each living in its own standalone library crate
under the modules/ directory. Each module:

  · Implements the TheaterModule trait from biospark-theater
  · Defines its WireType inputs and outputs
  · Can run standalone via `cargo run -p <module-name>`
  · Depends only on myth-wire and its own domain-specific libraries
  · Does NOT depend on any other module crate
  · Does NOT depend on Bevy unless the module is inherently spatial/visual
  · Does NOT share mutable state with any other module
  · Communicates exclusively via WirePackets through the Theater compositor

A module that cannot run standalone is not correctly designed. If `cargo run -p
terrain` fails without other modules running, the terrain module has an implicit
coupling that must be severed before development continues.

4.3  LAYER 3: OUTPUT ADAPTERS

Output adapters are the observation layer. They read what the simulation produced
and present it to a human or downstream system. They have no write access to the
narrative engine — they cannot change world state, only observe it.

The biospark-theater crate is the compositor library. It manages Channels (named
data streams), ChannelState, the TheaterModule and OutputHandler traits, and the
EngineTransport trait with its TcpTransport implementation. All renderer-specific
deps (bevy, egui, cpal) live behind Cargo feature gates in this crate only.

  [features]
  default = []
  bevy  = ["dep:bevy"]
  egui  = ["dep:egui", "dep:eframe"]
  audio = ["dep:cpal"]

A project that only needs prose output activates only the egui feature. No Bevy
is compiled. No audio. Only what is needed. Feature gate discipline is what keeps
the engine fast and the dep tree clean.

5.1  CRATE OWNERSHIP RULES

myth-wire
  ✓ WireType enum (17 variants — never add more without a formal decision)
  ✓ WirePacket, WirePayload (must be Serialize + Deserialize — always)
  ✓ BDna (Vec<bool>, exactly 64 elements — this invariant never relaxes)
  ✓ ModuleId, ChannelId (typed newtypes, not bare strings)
  ✗ NO logic, NO simulation, NO rendering
  ✗ NO async runtime
  ✗ deps beyond serde + bincode are PROHIBITED

mythos
  ✓ World state types
  ✓ Actor simulation logic (instinct, memory, social drive, dialogue)
  ✓ Genesis Container hierarchy (GenesisContainer → Capsule)
  ✓ Octave Capacity Law enforcement
  ✓ Tick loop / simulation step trait
  ✗ NO Bevy, NO egui, NO window, NO renderer of any kind
  ✗ NO async runtime (std channels only)
  ✗ NO module-specific logic (modules live in modules/)

Module crates (e.g., terrain, loom)
  ✓ Implements TheaterModule trait from biospark-theater
  ✓ Defines its WireType inputs and outputs
  ✓ Can run standalone: cargo run -p terrain
  ✓ deps: myth-wire + domain-specific libs
  ✗ Does NOT depend on other module crates
  ✗ Does NOT depend on Bevy unless inherently spatial/visual
  ✗ Does NOT share mutable state with other modules

biospark-theater
  ✓ Channel, ChannelState, compositor logic
  ✓ TheaterModule + OutputHandler traits
  ✓ EngineTransport trait + TcpTransport impl
  ✓ Renderer deps behind Cargo features ONLY
  ✗ Does NOT run the simulation
  ✗ Does NOT own world state
  ✗ Routes by WireType only — payload content is the handler's concern

Binaries (engine, instruments, theater-bin)
  ✓ 30–100 lines of wiring
  ✓ Imports library crates, connects them, starts the process
  ✗ Contains NO business logic
  ✗ Contains NO simulation code
  ✗ Contains NO rendering code

5.2  PUBLISHING PROGRESSION

Development uses workspace path deps:

  [dependencies]
  myth-wire = { path = "../myth-wire" }

When the API stabilizes, promote to git deps:

  myth-wire = { git = "...", tag = "v0.1" }

Only publish to crates.io when the API is genuinely stable. Premature publishing
followed by breaking changes is a project debt that compounds. Stay in path deps
until stability is confirmed.

CHAPTER 6  ·  Process Model & TCP Communication
────────────────────────────────────────────────────────────────────────────────

The three binaries are three separate OS processes. They communicate over TCP.
Always. Even on localhost. This is not a performance decision — it is an
architectural discipline.

  engine          binds  127.0.0.1:7700     headless, no window
  instruments     client  egui rack UI       sends CTL packets, receives state
  theater-bin     client  compositor         receives WirePackets, renders

The discipline of using a network boundary guarantees that the separation between
simulation and renderer is real, not accidental. Moving from localhost to a home
network is a config line change. Moving to the cloud is a config line plus a TLS
wrapper. Nothing in the simulation logic changes.

Arc<Mutex<>> across process boundaries defeats the purpose entirely. Shared memory
between these three processes is prohibited.

The EngineTransport trait abstracts the connection mechanism:

  · Local: TcpTransport on 127.0.0.1:7700
  · Network: TcpTransport with remote IP + TLS wrapper
  · Test: MockTransport that collects WirePackets in memory

Always program against the EngineTransport trait. Never against TcpTransport
directly. The abstraction layer is what makes the test harness work.

CHAPTER 7  ·  Wire Type System
────────────────────────────────────────────────────────────────────────────────

Wire types are the language of the system. Every connection between nodes — every
WirePacket emitted by a module, every edge in the Harmonic Tensor Graph — carries
exactly one wire type. There are 17 canonical wire types. This number is fixed.
Adding a new wire type requires a formal architectural decision, not a code change.

  #   Symbol   Full Name       Carries
  ─── ──────── ─────────────── ──────────────────────────────────────────────────
  01  DAT      DATA            Typed JSON payloads — universal fallback for any
                               data that doesn't fit a more specific type.
  02  CTL      CONTROL         Boolean, gate, and trigger signals. Used for
                               activating modules, opening gates, switching states.
  03  AUD      AUDIO           Waveform and sample streams.
  04  NAR      NARRATIVE       Story text, prose, dialogue, and lore payloads.
  05  TMP      TEMPORAL        Time signals: tick counts, clock data, timestamps.
  06  AGT      AGENT           Agent instruction packets and state streams.
  07  VIS      VISUAL          Image, render, and shader streams.
  08  SPA      SPATIAL         3D coordinates, voxel data, hex grid data.
  09  BHV      BEHAVIORAL      Emotion signals, drive values, decision packets.
  10  SOC      SOCIAL          Relationship maps, faction data, reputation values.
  11  ENR      ENERGY          Power levels, resource flows, capacity values.
  12  IDN      IDENTITY        B-DNA records, lineage chains, covenant data.
  13  EVT      EVENT           Cosmic bus events — world-level broadcast signals.
  14  AST      ASSET           File references, binary blobs, media URIs.
  15  MET      META            Schema definitions, type structures, manifest data.
  16  LGC      LOGIC           Boolean expressions, rule payloads, condition trees.
  17  RES      RESONANCE       Resonance field values — the gravity of the graph.
                               New. Every node that participates in the Condition
                               Engine must support at least one RES input.

DAT is the universal fallback. When data doesn't clearly belong to a more specific
type, use DAT — and then evaluate during the next architecture review whether a
dedicated type is warranted.

RES (Resonance) is the newest addition. It carries field values through the Harmonic
Tensor Graph, enabling resonance-based gate evaluation and dynamic edge formation
in the Condition Engine. Any node that contributes to or is influenced by world
resonance must declare a RES input.

7.1  TYPE ENFORCEMENT

Wire types are strictly enforced. A connection between a NAR output and an AUD
input is a type error. The router in biospark-theater enforces this at connection
time — mismatched types do not produce silent data corruption, they produce a
hard error at wire-up.

Conversion between types requires an explicit conversion node. A NAR→TTS→AUD
pipeline (text to audio) is valid because it passes through a declared TTS node.
There is no implicit coercion.

7.2  THE BDna TYPE

Every Capsule, every Container, every agent in the system carries a BDna field.
In Rust:

  struct BDna(Vec<bool>);

The Vec<bool> must be exactly 64 elements. This invariant never relaxes. Shorter
or longer arrays are rejected at deserialization time. The 64 positions encode
the complete lineage position of the entity in the world hierarchy — its place
in the cosmic order as a binary signature.

In Rust this is a fixed-size array — the compiler enforces the 64-element
invariant at the type level, no runtime validation needed:

  bdna: [bool; 64],  // exactly 64 positions — compile-time guarantee

CHAPTER 8  ·  B-DNA Lineage System
────────────────────────────────────────────────────────────────────────────────

B-DNA (Biographical DNA) is the provenance system of the Quantum Quill. It answers
three questions for every entity in the system:

  1. Where did this come from?
  2. What has it evolved into?
  3. What rules follow it wherever it goes?

8.1  THE B-DNA RECORD

  #[derive(Debug, Clone, Serialize, Deserialize)]
  pub struct BDnaRecord {
      pub signature:   String,                  // 64-bit binary string derived from lineage
      pub generation:  u32,                     // 0 = original, 1+ = derivative
      pub parent_ids:  Option<Vec<String>>,     // IDs of direct parent entities
      pub root_id:     Option<String>,          // ID of the original generation-0 entity
      pub covenant:    Option<CovenantRecord>,  // binding rules that follow this entity
      pub metadata:    Option<BDnaMeta>,
      pub sealed:      bool,                    // if true, covenant and parent_ids are immutable
  }

  #[derive(Debug, Clone, Serialize, Deserialize)]
  pub struct BDnaMeta {
      pub forked_from: Option<String>,          // if this entity was forked from another
      pub created_by:  Option<String>,          // creator identity
  }

  #[derive(Debug, Clone, Serialize, Deserialize)]
  #[serde(rename_all = "snake_case")]
  pub enum CovenantScope {
      Local,          // applies only to this entity
      Inherited,      // applies to all descendants
      CrossContext,   // follows the entity across any Genesis Container
  }

  #[derive(Debug, Clone, Serialize, Deserialize)]
  pub struct CovenantBinding {
      pub rule:        String,         // the binding rule as a plain-English string
      pub scope:       CovenantScope,
      pub enforceable: bool,           // true = mechanical enforcement; false = narrative
  }

  #[derive(Debug, Clone, Serialize, Deserialize)]
  pub struct CovenantRecord {
      pub bindings: Vec<CovenantBinding>,
  }

8.2  GENERATION SEMANTICS

Generation 0 is the original. It has no parentIds and no rootId. It is its own
root. Every derivative sets generation to parentGeneration + 1 and sets rootId
to the generation-0 ancestor's ID.

When an entity is forked — a character variant, a parallel timeline event, an
alternate module configuration — the fork records the forkedFrom field in metadata,
increments the generation counter, and carries forward any covenant bindings that
have scope 'inherited' or 'cross-context'.

8.3  COVENANT BINDING

Covenants are the rules that follow an entity. A covenant binding with scope
'local' applies only to that entity. A binding with scope 'inherited' applies
to all descendants. A binding with scope 'cross-context' applies to the entity
and any entity that references it, regardless of which Genesis Container they
inhabit.

Covenants with enforceable: true are enforced mechanically — the engine will
reject operations that violate them. Covenants with enforceable: false are
narrative covenants — they are honored by agents and storytellers but not
enforced by the engine. This distinction matters for game design: narrative
covenants create dramatic tension. Mechanical covenants create hard system rules.

8.4  THE LINEAGE HASH REQUIREMENT

No Capsule may be inserted into a sealed Genesis Container without a lineage_hash
field in its BDna record. This is a hard insertion rule enforced by assert_capacity
and the Vault submission pipeline. An entity without provenance is not an entity —
it is noise.

CHAPTER 9  ·  Vault Overview & Data Model
────────────────────────────────────────────────────────────────────────────────

The Vault is the delivery and publication layer of the Quantum ecosystem. Where the
Genesis Container is the creative workspace — the sealed physics of a living world —
the Vault is the archive and distribution system for artifacts produced within it.

The Vault stores everything that has completed its creative lifecycle and is ready
for use by other projects, agents, or human collaborators. It is the bridge between
creation and consumption.

9.1  VAULT DATA MODEL

Five primary entity types govern the Vault:

BLUEPRINT
  A complete world specification, software module definition, or narrative framework.
  The root entry point. A Blueprint can be submitted, validated, reviewed, and
  approved. Once approved, it may be modularized into Modules and assembled into
  Tomes.

  {
    id:       string,    // unique Blueprint ID
    title:    string,    // human-readable name
    author:   string,    // creator identity
    version:  string,    // semver string
    status:   'draft' | 'pending' | 'approved' | 'rejected' | 'archived'
  }

MODULE
  A self-contained component derived from a Blueprint. Modules are the atomic
  reusable units of the Vault — a single character profile, a single software
  component, a single audio patch, a single visual asset. Modules can be tagged,
  versioned, searched, and assembled into Collections.

  {
    id:            string,
    blueprintId:   string,      // parent Blueprint
    componentType: string,      // the type of component this module represents
    tags:          string[],    // searchable tags
    dependencies:  string[]     // IDs of other Modules this depends on
  }

COLLECTION
  A curated group of Modules assembled for a specific purpose. Collections are
  the building blocks of Tomes. A Collection might group all the character modules
  for Act I of a story, or all the UI components for a software release.

TOME
  A compiled artifact ready for distribution. A Tome assembles one or more
  Collections into a single deliverable — a complete story arc export, a software
  release package, a music album, or a world seed package. Tomes include a
  compiledArtifactURL pointing to the packaged output.

  {
    id:                   string,
    title:                string,
    collectionIds:        string[],
    compiledArtifactURL:  string
  }

REVISION
  A versioned snapshot of a Blueprint or Module. Revisions track what changed,
  when, and why. They are the time axis of the Vault — the full history of how
  an artifact evolved from submission to publication.

9.2  VAULT WORKFLOW

The Vault pipeline moves every Blueprint through a defined sequence of stages:

  Submit → Validate → Review → Modularize → Register → Assemble → Publish

  1. SUBMIT      Author submits a Blueprint via CLI or API.
  2. VALIDATE    Automated validation checks wire type consistency, B-DNA
                 requirements, and structural integrity.
  3. REVIEW      Human or agent review. Approves or rejects with feedback.
  4. MODULARIZE  Approved Blueprint is broken into constituent Modules.
                 Each Module gets its own ID, tags, and dependency list.
  5. REGISTER    Modules are registered in the Vault index for search and discovery.
  6. ASSEMBLE    Modules are selected into Collections. Collections are composed
                 into Tomes.
  7. PUBLISH     Tome is compiled and its compiledArtifactURL is set. The Tome
                 becomes discoverable and downloadable.

CHAPTER 12  ·  Genesis Container Architecture
────────────────────────────────────────────────────────────────────────────────

The Genesis Container is the sealed universe of a BioSpark project. Everything
inside it obeys the Octave Capacity Law — a power-of-2 recursive depth system.
Containers declare their growth behavior at seeding and seal at whatever depth
they actually reach.

12.1  FOUR HIERARCHY LEVELS

The hierarchy has exactly four levels. The level names are stable. The capacity
at each level is determined by the declared or reached octave depth of that
specific container instance — not a universal fixed number.

  Level 0   GENESIS CONTAINER    The sealed universe / project root
  Level 1   MYTHOS CONTAINER     Major domain, arc, collection, or department
  Level 2   CONTAINER            Sub-collection, chapter, module, feature
  Level 3   CAPSULE              Atomic unit — one entity, track, item, component

Each level instance has its own growth mode and octave. A Mythos Container at
octave 4 holds up to 16 Containers. The same Mythos Container at octave 5 holds
32. The law is always relative to the declared octave — there is no globally
fixed number of children at any level.

12.2  CANONICAL DEFAULT: OCTAVE 4

Octave 4 (16 children) is the canonical default for most containers. This is not
arbitrary — it aligns with:

  · 16-bit audio depth (standard digital audio)
  · 16-bar musical phrases (standard song structure)
  · 16-channel MIDI mixing (General MIDI standard)
  · 2^4 — a natural power-of-two memory allocation size

Departing from octave 4 is a deliberate architectural decision that must be
declared at seeding time. Departing without declaration is a protocol violation.

12.3  DOMAIN MAPPINGS

The four-level hierarchy applies universally across all BioSpark project types:

  Narrative / Worldbuilding:
    Genesis  → Story World / Universe
    Mythos   → Story Arc / Region / Era
    Container → Scene Collection / Culture / Faction
    Capsule  → Character / Event / Object / Lore Entry
    (Dynamic growth recommended — worlds grow organically)

  Music & Discography:
    Genesis  → Artist / Band / Musical Persona
    Mythos   → Album Series / Season / World Arc
    Container → Album / EP / Compilation
    Capsule  → Individual Track / Song
    (Fixed octave 4 for Albums. Dynamic for Artist Genesis Container.)

  Software / Application:
    Genesis  → Application / Platform
    Mythos   → Module / Department
    Container → Feature Set / Sub-system
    Capsule  → Component / Function / Node
    (Fixed octave 4 recommended. Module boundaries are intentional constraints.)

  Agent Systems:
    Genesis  → Simulation / World
    Mythos   → Guild / Department
    Container → Squad / Project
    Capsule  → Individual Actor / Tool
    (Dynamic for Actor containers. Fixed for Guild/Department.)


────────────────────────────────────────────────────────────────────────────────
CHAPTER 13  ·  Octave Capacity Law
────────────────────────────────────────────────────────────────────────────────

The Octave Capacity Law is the mathematical spine of the container system. It
replaces any earlier fixed-number scheme. All capacity exists on a binary recursive
scale — the same scale that governs audio octaves, binary memory addressing, and
power-of-two resonance harmonics.

  Octave 0  →    1    The Prime Seal / root law / unified physics
  Octave 1  →    2    Duality — polarity, signal/noise, on/off
  Octave 2  →    4    Quadrants — cardinal anchors, four states
  Octave 3  →    8    First musical octave — melody emerges from rhythm
  Octave 4  →   16    CANONICAL DEFAULT — where most containers rest
  Octave 5  →   32    Extended depth — complex systems, sprawling worlds
  Octave 6  →   64    Deep systems — full creative ecosystems
  Octave 7  →  128    Maximum practical depth — rare, Genesis-scale only

Note: Earlier design documents may reference a "12-12-12" or "1,728 capsule"
scheme. Those documents are outdated. The Octave Capacity Law supersedes all
prior fixed-number specifications. There is no system-wide cap of 1,728 capsules
or any other fixed number.

13.1  OCTAVE BEHAVIOR MODES

Beyond the linear Expand mode, containers can be configured to behave differently:

  Expand   — Linear: grows deeper, multiply by 2 per octave (standard)
  Bounce   — Reverse: 64 → 32 → 16 → 8 → ... used for decay, entropy arcs
  Loop     — Cyclic: hits floor and starts next octave up, for cyclic systems
  Fade     — Weighted: contribution weight reduces with depth, like volume envelope

Behavior mode is a seeding decision. It is declared once and governs the container
for its entire lifecycle.


────────────────────────────────────────────────────────────────────────────────
CHAPTER 14  ·  Growth Modes: Fixed vs. Dynamic
────────────────────────────────────────────────────────────────────────────────

Every container declares a growth mode at seeding. This is the most architecturally
significant decision made about a container, after octave depth.

14.1  FIXED (VHD-STYLE)

Octave depth is declared at seeding. That declaration is the ceiling AND the law
from day one. The container pre-allocates its shape. Validation is fast. Creative
constraints are enforced intentionally — like choosing a canvas size before painting.

  {
    "growth_mode": "fixed",
    "declared_octave": 4,
    "max_children": 16
  }

Best for: World Genesis Containers, sealed physics systems, module definitions,
any container where the boundary is the creative point.

14.2  DYNAMIC (VHDX-STYLE)

Starts at octave 1 and grows as content is added, expanding to the next octave
automatically when it fills. Shape emerges from use — it mirrors how creative work
actually develops.

An optional estimated ceiling can be set, adjusted, and re-estimated while the
container is active. At seal time the ceiling locks to actual depth reached. The
estimate becomes historical metadata — it records the story of how the creator
understood the work as it grew.

  {
    "growth_mode": "dynamic",
    "estimated_ceiling_octave": 4,
    "current_octave": 2,
    "ceiling_adjustments": 1,
    "max_children": null    // null until sealed
  }

Best for: Actor/Character containers, living narrative systems, artist discographies,
anything that grows organically over time.

The baby analogy: You might estimate 8 pounds but adjust to 12 as reality develops.
At birth (seal), whatever weight the child actually is becomes the locked truth.
The estimate is the ultrasound. The sealed value is the birth record.

A container that consistently exceeded its estimated ceiling tells a different story
than one that sealed at half the estimate. The agent layer uses this metadata to
reason about world health over time.

14.3  SEALED STATE

Regardless of growth mode, a sealed container is structurally immutable. No new
Capsules can be added. No Containers can be removed. The hierarchy is frozen.

Capsule payloads (the content of individual Capsules) may still be updated after
sealing. Hierarchy is frozen. Content is not.

  {
    "growth_mode": "dynamic",
    "estimated_ceiling_octave": 4,
    "sealed_octave": 6,         // grew past estimate — that's information
    "ceiling_adjustments": 2,   // estimated twice, still underestimated
    "sealed_at": 1746321600000
  }

CHAPTER 16  ·  Capacity Enforcement & Insertion Rules
────────────────────────────────────────────────────────────────────────────────

16.1  THE assert_capacity FUNCTION

Before every insert, the assert_capacity guard runs on the target container.
Returns Ok(()) if insertion is allowed, expanding octave if dynamic.
Returns Err if the container is full and growth_mode is Fixed.

  /// Capacity guard — enforces current octave limit.
  /// Call before every capsule or child container insertion.
  pub fn assert_capacity(cap: &mut CapacityMetadata, label: &str) -> Result<(), String> {
      let limit = 2u32.pow(cap.current_octave as u32);
      if cap.child_count >= limit {
          match cap.growth_mode {
              GrowthMode::Dynamic => {
                  // Expand to next octave automatically
                  cap.current_octave += 1;
                  cap.max_children_current = 2u32.pow(cap.current_octave as u32);
                  if let Some(ceiling) = cap.estimated_ceiling_octave {
                      if cap.current_octave > ceiling {
                          cap.ceiling_adjustments += 1;
                          // Caller should log a ceiling revision event
                      }
                  }
                  Ok(())
              }
              GrowthMode::Fixed => Err(format!(
                  "Capacity violation: {} is full at octave {} ({} children). \
                   Fixed mode — create a new parent container.",
                  label, cap.current_octave, limit
              )),
          }
      } else {
          Ok(())
      }
  }

16.2  SEVEN INSERTION RULES

  1. Call assert_capacity() on the target container before every insert.

  2. Fixed mode overflow: The container is full. Create a new sibling container
     under the same or a new Mythos. Do not exceed the declared octave.

  3. Dynamic mode overflow: The container expands to the next octave automatically.
     If expansion would exceed estimated_ceiling_octave, log a ceiling revision
     event and increment ceiling_adjustments.

  4. Sealed container: No structural changes permitted after seal. Capsule payload
     updates are allowed. Hierarchy is frozen.

  5. B-DNA requirement: Every Capsule must have a lineage_hash. Nothing without
     provenance enters a sealed Genesis Container.

  6. Resonance requirement: Every container must declare a resonance_hz at seeding.
     Lesser Seals must declare a harmonic_ratio to their parent seal frequency.

  7. Vault storage: Any asset not in active use must be stored in the Vault, not
     retained in the active Genesis Container's working memory.

16.3  GREATER SEAL vs. LESSER SEAL

Greater Seal — the root container of the entire metaverse. No declared size. Grows
until resonance can no longer sustain coherent expansion. Exactly one per metaverse.
Cannot be nested inside anything. Has no parent_seal_id.

Lesser Seal — every other sealed container. Has declared growth mode and optional
estimated ceiling. Can nest inside other Lesser Seals if resonance is harmonically
compatible. A Lesser Seal cannot contain a Greater Seal. This is a hard physics law.

Every Lesser Seal must declare a harmonic_ratio to its parent seal frequency. The
harmonic relationship between nested seals determines their compatibility in the
Harmonic Tensor Graph and governs resonance-based gate conditions between them.

CHAPTER 20  ·  LLM Routing — Ollama Local-First Architecture
────────────────────────────────────────────────────────────────────────────────

All LLM inference in the Quantum Quill system defaults to local models via Ollama.
The routing logic follows a strict precedence:

  Local Ollama → Available? → Use it.
        ↓ Not available
  Check internet connection → Available? → Use configured cloud fallback.
        ↓ Not available
  Return cached response if one exists for this query.
        ↓ No cache
  Return degraded-mode response (inform user of limited functionality).

This routing ensures the system is maximally useful offline and on low-bandwidth
connections, while being able to leverage cloud models for tasks that exceed local
capacity.

The faction conditioning system (Chapter 34) injects faction identity into every
LLM system prompt, regardless of which provider is serving the request. The prompt
injection is provider-agnostic.

────────────────────────────────────────────────────────────────────────────────

TOKEN    A transactional or temporary symbol. Represents resources, permissions, or

SIGIL    A unique personal binding. Every actor has exactly one Sigil — derived from
         their B-DNA signature and unreplicable. When two actors interact, their Sigils

      /// Build the LLM system prompt for this faction.
      pub fn system_prompt(&self) -> String {
          let tier = if self.energy_tier >= 0 {
              format!("+{}", self.energy_tier)

              self.energy_tier.to_string()

              "You are operating in the {} faction register.\n\
               Emotional register: {}.\n\
               Use language evoking: {}.\n\
               Visual palette: {}. Energy tier: {}.",
              self.name,
              self.emotional_register.join(", "),
              self.prompt_keywords.join(", "),
              self.primary_color,
              tier,
          )
      }
  }

CHAPTER 41  ·  LoomEngine — 5-Stage Pipeline
────────────────────────────────────────────────────────────────────────────────

The LoomEngine transforms discrete world events (EVT packets from the Chronicle)
into flowing narrative output (NAR packets ready for export). The five stages are
named after physical loom components — the tool that weaves threads into cloth
is the metaphor for weaving events into story.

  STAGE 1  WARP THREADS
    Input:   Raw EVT packets from the Chronicle module
    Process: Classify events as PLOT_CRITICAL, CHARACTER_MOMENT, WORLD_DETAIL,
             or BACKGROUND. Only first two types proceed. WORLD_DETAIL is indexed
             but not forwarded. BACKGROUND is discarded — simulation noise, not story.

  STAGE 2  PATTERN HARNESSES
    Input:   Classified narrative events
    Process: Group events into causal chains. Identify which events caused which
             others, which characters are affected, which chains are parallel vs
             intersecting. Output is a structured event graph, not a list.

  STAGE 3  OPERATION SHUTTLE
    Input:   Event graph
    Process: Make editorial decisions — pacing, what to show now vs later, when
             to reveal what. Applies the Story module's beat structure and tension
             meter to determine output selection.

  STAGE 4  REED MERGER
    Input:   Selected events
    Process: Merge parallel event threads into coherent linear (or branching)
             narrative sequence. Handles simultaneous events, off-screen events,
             and time-skipped periods.

  STAGE 5  TAKE-UP ROLL
    Input:   Merged sequence
    Process: Format as NAR wire output in requested format (screenplay, prose chapter,
             game event feed, summary digest). Apply faction aesthetic conditioning.
             Emit as NAR packets to Theater output adapter.

The HTG uses these frequencies to:
  · Form dynamic edges — nodes with harmonically compatible frequencies (2:1, 3:2,
    4:3 ratios) form natural edges. Dissonant ratios create resistance.
  · Evaluate gate conditions — checks whether cumulative resonance at a gate node
    has reached the threshold required to open it.
  · Calculate heraldic positions — a node's HTG position partially determines its
    heraldic ascent/descent trajectory.
  · Surface shadow elements — when resonance drops below threshold, bound shadow
    elements become active.

RES wire type carries resonance field values between nodes.

CHAPTER 46  ·  Trinary Validation Gate & Gate Conditions
────────────────────────────────────────────────────────────────────────────────

The TRINARY VALIDATION GATE (TVG) is the canonical gate evaluation structure.
All three conditions must be TRUE for a TVG to open:

  CONDITION 1 (Structural):   Correct hierarchy level?
  CONDITION 2 (Resonance):    RES wire value at gate ≥ declared threshold?
  CONDITION 3 (Heraldic):     Current heraldic type satisfies gate's requirement?

Five condition source types feed TVG evaluations:
  MIDI       — Physical MIDI controller input (CTL wire)
  B-DNA      — Lineage and covenant state (IDN wire)
  Heraldry   — Current heraldic position (MET wire)
  Voxel      — Spatial perception field values (SPA wire)
  Resonance  — HTG resonance field values (RES wire)

CHAPTER 47  ·  Shadow Elements & Resonance Thresholds
────────────────────────────────────────────────────────────────────────────────

Every HTG node has a resonance threshold — the minimum RES value below which
the node enters a degraded state. When a node's resonance drops below threshold:

  1. The node's bound_elements list is checked for shadow elements.
  2. Active shadow elements surface, modifying node behavior.
  3. RESONANCE_DROP EVT packet is emitted on the cosmic bus.
  4. Soralinde (the Observer) is notified via the Chronicle.

Shadow elements are the "dark side" of the Periodic Table of Thaumaturgy. Every
manifest element has a corresponding shadow element that activates under low-resonance
conditions — always the inversion or corruption of the manifest element's behavior.

Example: A Sylvanid "Verdant Growth" node that drops below threshold will have its
shadow element "Consuming Rot" become active. Growth mechanics are replaced with
decay mechanics until resonance is restored.


────────────────────────────────────────────────────────────────────────────────
CHAPTER 48  ·  Heraldic Ascent / Descent Mechanics
────────────────────────────────────────────────────────────────────────────────

Entities can ascend to higher heraldic types or descend to lower ones through the
actions they take and the resonance they accumulate or lose.

Ascent path:   Mark → Trait → Token → Sigil → Emblem → Device → Glyph → Crest → (Lesser Seal)
Descent path:  (Lesser Seal) → Crest → Glyph → Device → Emblem → Sigil → Token → Trait → Mark

Ascent requires:
  · Sustained high resonance (above threshold for N consecutive ticks)
  · Successful TVG evaluation at the ascent gate
  · A covenant binding that permits ascent

Descent occurs:
  · When resonance drops below threshold for M consecutive ticks
  · When a covenant binding is violated (enforceable: true)
  · When an Apex_Fall event fires and the entity is in strata[+2]

heraldic_birth is immutable. No matter how far an entity descends, the birth record
remains. This is the B-DNA of the heraldic system — you can change what you are,
but you cannot change what you were.

CHAPTER 50  ·  White-Light Handshake Protocol
────────────────────────────────────────────────────────────────────────────────

The White-Light Handshake governs how boundary crossings between Genesis Containers
occur safely (a character from one world appearing in another, a module being imported).

  1. DECLARE    Importing container declares intent — source container ID + entity B-DNA.
  2. VERIFY     CORE_SEAL_V1 system verifies both containers are authenticated and
                entity B-DNA is consistent between both records.
  3. COVENANT   Cross-context covenant bindings transfer intact. Local/inherited bindings
                remain in the source container.
  4. FORK       New B-DNA record created: generation = source + 1, parentIds = [source.id],
                rootId = source.bdna.rootId ?? source.id, forkedFrom = source.id.
  5. SEAL       Handshake recorded in Chronicle of both containers as CROSS_WORLD_IMPORT
                EVT packet. Immutable after this step.

The Instruments UI marks handshake completion with a white-light pulse.

CHAPTER 54  ·  Pre-Flight Checklist
────────────────────────────────────────────────────────────────────────────────

Before writing any new myth-os code:

  [ ] Which layer? engine (narrative) / module / adapter / binary
      Default to library crate. Only promote to binary if there's a main() reason.

  [ ] Zero renderer deps in the engine?
      cargo tree -p mythos | grep bevy
      Bevy appears → STOP. Revert. Fix before proceeding.

  [ ] Can this module run standalone?
      cargo run -p <module-name>
      Fails alone → coupling present. Fix it.

  [ ] Modules communicating via WirePackets only?
      Any module crate listed in another module's Cargo.toml → STOP.

  [ ] Renderer dep crept into a library crate?
      cargo tree -p mythos | grep "bevy\|egui\|cpal"

  [ ] Is the binary under 100 lines?
      main.rs > 100 lines → logic leaked into binary. Move to library.

  [ ] Three-Way Alignment on every new type?
      Structural position / Functional role / Symbolic type — all three.

  [ ] Every Capsule has a lineage_hash?
      No provenance = no insertion.

  [ ] Using EngineTransport, not TcpTransport directly?

  [ ] Every container declares resonance_hz at seeding?
      Lesser Seals must declare harmonic_ratio to parent.

CHAPTER 55  ·  Anti-Patterns to Avoid
────────────────────────────────────────────────────────────────────────────────

THE BEVY HOST TRAP
  Simulation logic in impl Plugin for X { fn build() }. Bevy systems are renderers.
  Fix: Logic in mythos or module crates. Bevy reads state. Bevy never writes simulation.

THE MONOLITH
  One binary loads everything.
  Fix: Each module is standalone. Each binary composes only what it needs.

MODULE COUPLING
  Module A imports Module B types and calls its functions.
  Fix: WirePackets only. Modules are strangers to each other.

SHARED MEMORY ACROSS PROCESSES
  Arc<Mutex<>> spanning what should be separate binaries.
  Fix: TCP. Always. Even localhost. The boundary is the point.

FAT BINARY
  All logic in main.rs. Library crate is empty scaffolding.
  Fix: Library crate holds everything. Binary is wiring only.

PREMATURE PUBLISHING
  Pushing to crates.io before the API is stable.
  Fix: Path deps during development. Publish when the API is genuinely stable.

RENDERER DEP CREEP
  A "small" renderer import sneaks into mythos or a module crate.
  Fix: cargo tree -p mythos | grep bevy after every change. Bevy appears → revert.

FIXED-NUMBER THINKING
  Designing containers with a fixed 12-or-16 child limit as a system-wide constant.
  Fix: All capacity is octave-relative. Default is octave 4 (16). Departure from
  octave 4 is a deliberate decision, not a violation.

CLOUD-FIRST INFERENCE
  Routing LLM calls to cloud before checking for local Ollama.
  Fix: Always check Ollama first. Cloud is a fallback, not the default.

MISSING B-DNA
  Creating a Capsule without a lineage_hash.
  Fix: No provenance, no insertion. Enforced at every insertion point.
